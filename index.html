责任链模式

发表于 2015-09-04   |   分类于 设计模式 ， 行为型设计模式   |  

责任链模式(ChainOfResponsibilityPattern)是一种行为型设计模式。意图：使多个对象都有机会处理请求，从而避免发送者和接收者之间的耦合关系。将所有可以处理请求的对象串成一条链，并沿着这条链传递请求，直到有对象处理它。

代码实现

请求对象

class Request
{
    private int _day;
    private string _reason;


    public int Day
    {
        get { return _day; }
        set { _day = value; }
    }

    public string Reason 
    {
        get { return _reason; }
        set { _reason = value; }
    }
}

一个抽象“链”对象

抽象对象定义抽象的处理请求的方法，并定义下个链对象。

abstract class AbstractHandler
{
    private AbstractHandler _successor;
    protected string _name;

    public AbstractHandler(string name)
    {
        _name = name;
    }

    public AbstractHandler Successor 
    {
        get { return _successor; }
        set { _successor = value; }
    }

    public abstract bool PassHandle(Request request);
}

具体的请求处理对象

具体的对象根据条件实现请求的方法。

class PM : AbstractHandler
{
    public PM(string name) :
        base(name)
    {

    }

    public override bool PassHandle(Request request)
    {
        if (request.Day < 0.5)
        {
            Console.WriteLine("PM " + _name + " 批准请假。 ");
            return true;
        }

        return Successor.PassHandle(request);
    }
}

class HR : AbstractHandler
{
    public HR(string name) :
        base(name)
    {

    }

    public override bool PassHandle(Request request)
    {
        if (request.Day > 0.5 && request.Day < 3) 
        {
            Console.WriteLine("HR " + _name + " 批准请假。 ");
            return true;
        }

        return Successor.PassHandle(request);
    }
}

class Manager : AbstractHandler
{
    public Manager(string name) :
        base(name)
    {

    }

    public override bool PassHandle(Request request)
    {
        if (request.Reason.Equals("正当理由"))
        {
            Console.WriteLine("Manager " + _name + " 批准请假。 ");
            return true;
        }

        Console.WriteLine("Manager " + _name + " 认为理由不正当，不允许批准。 ");
        return false;
    }
}

测试代码

class Program
{
    static void Main(string[] args)
    {
        PM pm = new PM("项目经理");
        HR hr = new HR("人力资源");
        Manager manager = new Manager("总经理");

        pm.Successor = hr;
        hr.Successor = manager;


        Request request1 = new Request();
        request1.Day = 2;
        request1.Reason = "正当理由";

        pm.PassHandle(request1);

        Request request2 = new Request();
        request2.Day = 6;
        request2.Reason = "不正当理由";

        pm.PassHandle(request2);

        Console.Read();
    }
}

个人理解

责任链模式通过一条链来处理请求，请求者并不知道处理者对象，分离了请求者和处理者。但是每个请求都必须从链头开始处理，造成了内存中的垃圾对象。

适用性

有多个对象处理请求时，而且请求在运行时候决定。

在不明确处理者的时候，向多个对象提出一个请求。

可处理一个请求的对象集合应该被动态指定。

如果存在N对N，或者是常规线性关系，我们完全可以用表驱动代替责任链。

责任链并非只可以在上下级间传递，也可以在同级间传递。

模板模式

发表于 2015-09-03   |   分类于 设计模式 ， 行为型设计模式   |  

模板模式(TemplatePattern)是一种行为型设计模式。意图：定义一个算法的框架，而将其中的一些步骤的实现延迟到子类中。

代码实现

抽象模板类

抽象模板类定义算法的框架，定义抽象行为，让其在子类中实现。

abstract class AbstractBeverage
{
    public void Create()
    {
        BoilWater();
        AddCoundiments();
        Brew();
        PourInCup();

        if (Hook())
        {
            Again();
        }
    }

    public void BoilWater()
    {
        Console.WriteLine("烧开水");
    }

    public abstract void AddCoundiments();

    public abstract void Brew();

    public void PourInCup()
    {
        Console.WriteLine("倒入杯子中");
    }

    protected virtual bool Hook()
    {
        return false;
    }

    private void Again()
    {
        Console.WriteLine("在来一杯");
    }
}

具体实现的子类

子类实现抽象函数中的方法，实现个性化。

class Tea : AbstractBeverage
{
    public override void AddCoundiments()
    {
        Console.WriteLine("加入茶叶");
    }

    public override void Brew()
    {
        Console.WriteLine("冲泡茶叶");
    }


    protected override bool Hook()
    {
        return true;
    }
}

class Coffer : AbstractBeverage
{
    public override void AddCoundiments()
    {
        Console.WriteLine("加入咖啡豆");
    }

    public override void Brew()
    {
        Console.WriteLine("冲泡咖啡");
        Season();
    }

    private void Season()
    {
        Console.WriteLine("加入2块方糖，调节味道");
    }
}

测试代码

class Program
{
    static void Main(string[] args)
    {
        AbstractBeverage beverageTea = new Tea();
        beverageTea.Create();

        AbstractBeverage beverageCoffer = new Coffer();
        beverageCoffer.Create();

        Console.Read();
    }
}

个人理解

模板方法通过定义一个相同的算法框架，而将其中部分步骤的实现延迟到子类中实现。

适用性

一次性实现算法的固定部分，将可变的行为留给子类。

各个子类中的公有行为被提取到超类中，提高了代码的可复用性。

通过钩子（hook）控制了子类的扩展。子类可以实现也可以不是先钩子方法。

解释器模式

发表于 2015-09-02   |   分类于 设计模式 ， 行为型设计模式   |  

解释器模式(InterpreterPattern)是一种行为型设计模式。意图：给定一个语言，定义它的文法的一种表示，并定义一个解释器，这个解释器使用该表示来解释语言中的句子。

代码实现

抽象解释器

抽象解释器声明一个所有具体的表达式所要实现的接口。

abstract class Expression
{
    public abstract int Interpreter(Dictionary<string, int> var);
}
终结符表达式

终结符表达式实现与文法中相关联的解释操作。

class VarExpression : Expression
{
    private string _key;
    public VarExpression(string key)
    {
        _key = key;
    }

    public override int Interpreter(Dictionary<string, int> var)
    {
        return var[_key];
    }
}
非终结符表达式

文法中每条规则对应一个非终结符表达式。非终结符表达式一般是指“+”等这种。

abstract class SymbolExpression : Expression
{
    protected Expression _left;
    protected Expression _right;

    public SymbolExpression(Expression left, Expression right)
    {
        _left = left;
        _right = right;
    }
}


class AddExpression : SymbolExpression
{
    public AddExpression(Expression left, Expression right) :
        base(left, right)
    {
    }

    public override int Interpreter(Dictionary<string, int> var)
    {
        return base._left.Interpreter(var) + base._right.Interpreter(var);
    }
}

class SubExpression : SymbolExpression
{
    public SubExpression(Expression left, Expression right) :
        base(left, right)
    {
    }

    public override int Interpreter(Dictionary<string, int> var)
    {
        return base._left.Interpreter(var) - base._right.Interpreter(var);
    }
}
封装的解释器
class Calculator
{
    private Expression _expression;
    private Dictionary<string, int> _dictionary;
    public Calculator(string str)
    {
        string[] listStr = str.Split(new Char[] { ' ' }, StringSplitOptions.RemoveEmptyEntries);
        GetDictonary(listStr);

        Expression left = null;
        Expression right = null;
        Expression expression = null;

        for (int i = 0; i < listStr.Length; i++)
        {
            switch (listStr[i])
            {
                case "+":
                    left = expression;
                    right = new VarExpression(listStr[++i]);
                    expression = new AddExpression(left, right);
                    break;
                case "-":
                    left = expression;
                    right = new VarExpression(listStr[++i]);
                    expression = new SubExpression(left, right);
                    break;
                default:
                    expression = new VarExpression(listStr[i]);
                    break;
            }
        }

        _expression = expression;
    }

    public int Run()
    {
        return this._expression.Interpreter(_dictionary);
    }


    private void GetDictonary(string[] listStr )
    {
        _dictionary = new Dictionary<string, int>();

        foreach (string curString in listStr)
        {
            if (curString != "+" && curString != "-")
            {
                _dictionary.Add(curString, Convert.ToInt32(curString));
            }
        }
    }
}
测试代码
class Program
{
    static void Main(string[] args)
    {
        string str = "1234 + 234234 - 948 + 3242 - 4994 + 3";
        Calculator calculator = new Calculator(str);
        Console.WriteLine(calculator.Run());

        Console.Read();
    }

}

个人理解

解释器是一个语法分析工具。

适用性

当一个语言需要解释执行，你可以将语言中的句子表示为抽象的语法树时。

有一个简单的语法规则（如：sql）,一些重复的操作（如：+ - * /）。

优缺点

增加一个规则只需要增加一个非终结符表达式。修改一个规则只需要修改非终结符表达式。具有良好的扩展性。

解释器模式会引起类的膨胀。每个语法都会产生一个非终结符表达式。而且进行语法解释的时候，会使用回调，影响效率。

代理模式

发表于 2015-08-30   |   分类于 设计模式 ， 创建型设计模式   |  

代理模式(ProxyPattern)是一种创建型设计模式。意图：为其他对象提供一种代理以控制对这个对象的访问。

代码实现

普通代理

接口类

提供公有的方法

public interface IGamePlayer
{
    void Login(string user, string password);

    void KillBoss();

    void Update();

    IGamePlayer GetProxy();
}
实例类

完成接口的方法

public class GamePlayer : IGamePlayer
{
    private string _user;

    public void Login(string user, string password)
    {
        _user = user;
        Console.WriteLine("user is " + user + " password is " + password);
    }

    public void KillBoss()
    {
        Console.WriteLine(_user + " kill a boss.");
    }

    public void Update()
    {
        Console.WriteLine(_user + " update one. ");
    }

    public IGamePlayer GetProxy()
    {
        return null;
    }
}
代理类
public class Proxy : IGamePlayer
{
    IGamePlayer _player;

    public Proxy(IGamePlayer player)
    {
        _player = player;
    }

    public void Login(string user, string password)
    {
        _player.Login(user, password);
    }

    public void KillBoss()
    {
        _player.KillBoss();
    }

    public void Update()
    {
        _player.Update();
    }

    public IGamePlayer GetProxy()
    {
        return this;
    }
}
测试代码
class Program
{
    static void Main(string[] args)
    {
        IGamePlayer player = new GamePlayer();
        IGamePlayer proxy = new Proxy(player);

        proxy.Login("test", "****");
        proxy.KillBoss();
        proxy.Update();

        Console.Read();
    }
}

强制代理

一般都是通过代理找到真实对象，强制代理却通过真实对象找到代理。

实体类

强制代理 只需要将实例类中的方法加入获得代理。 代理必须是真实对象，不能是外部new出来的。

public class ForceGamePlayer : IGamePlayer
{
    private string _user;
    private IGamePlayer _proxy = null;

    public IGamePlayer GetProxy()
    {
        _proxy = new Proxy(this);
        return _proxy;
    }

    public void Login(string user, string password)
    {
        _user = user;

        if (IsProxy())
        {
            Console.WriteLine("user is " + user + " password is " + password);
        }
        else
        {
            Console.WriteLine(" please use proxy. ");
        }

    }

    public void KillBoss()
    {
        if (IsProxy())
        {
            Console.WriteLine(_user + " kill a boss.");
        }
        else
        {
            Console.WriteLine(" please use proxy. ");
        }

    }

    public void Update()
    {
        if (IsProxy())
        {
            Console.WriteLine(_user + " update one. ");
        }
        else
        {
            Console.WriteLine(" please use proxy. ");
        }

    }

    public bool IsProxy()
    {
        if (null == _proxy)
        {
            return false;
        }
        else
        {
            return true;
        }
    }
}
测试代码
class Program
{
    static void Main(string[] args)
    {
        IGamePlayer forcePlayer = new ForceGamePlayer();
        IGamePlayer forceProxy = forcePlayer.GetProxy();
        forceProxy.Login("forcePlayer", "****");
        forceProxy.KillBoss();
        forceProxy.Update();

        Console.Read();
    }
}

有个性的代理

代理不仅可以完成主体接口，也可以实现其他接口不能完成的任务。

实体类

实体类可以继承多个接口，实现个性化的接口。

public class PersonalGamePlayerProxy : IGamePlayer, IProxy
{
    IGamePlayer _player;

    public PersonalGamePlayerProxy(IGamePlayer player)
    {
        _player = player;
    }

    public IGamePlayer GetProxy()
    {
        return this;
    }

    public void Login(string user, string password)
    {
        _player.Login(user, password);
    }

    public void KillBoss()
    {
        _player.KillBoss();
    }

    public void Update()
    {
        Money();
        _player.Update();
    }

    public void Money()
    {
        Console.WriteLine(" update on need $15. ");
    }
}
新增加的接口类
interface IProxy
{
    void Money();
}
测试代码
class Program
{
    static void Main(string[] args)
    {
        IGamePlayer personalPlayer = new GamePlayer();
        IGamePlayer personalProxy = new PersonalGamePlayerProxy(personalPlayer);
        personalProxy.Login("personalProxy", "****");
        personalProxy.KillBoss();
        personalProxy.Update();

        Console.Read();
    }
}

虚拟代理

在需要的时候才判断是否创建对象，可以避免代理对象较多引起的初始化缓慢的问题，缺点是要在每个方法中判断时候对象是否为null。

公用接口类
public interface Subject
{
    void Request();
}
实例类
public class RealSubject : Subject
{
    public void Request()
    {
        Console.WriteLine(" real subject ");
    }
}
虚拟代理类
public class VirtualProxy : Subject
{
    private Subject _subject;

    public void Request()
    {
        if (null == _subject)
        {
            _subject = new RealSubject();
        }

        _subject.Request();
    }
}
测试代码
class Program
{
    static void Main(string[] args)
    {
        VirtualProxy virtualProxy = new VirtualProxy();
        virtualProxy.Request();

        Console.Read();
    }
}

个人理解

在代理模式中，真实的角色实现业务逻辑，而不关心事务，事务由代理对象来处理。真实对象，只要实现了接口，不管怎么变化，代理类都不需要变化。

适用性

在要用比较通用和复杂的指针代替简单的指针的时候。

proxy并不一定要求保持接口的一致性，只要能够实现间接控制，有时候损及一些透明性是可以接受的。

“增加一层间接层”是软件系统中对许多负责问题的一种常见解决方法。

适配器Adapter 为它所适配的对象提供了一个不同的接口。相反，代理提供了与它的实体相同的接口。

尽管Decorator的实现部分与代理相似，但Decorator的目的不一样。Decorator为对象添加一个或多个功能，而代理则控制对对象的访问。

享元模式

发表于 2015-08-26   |   分类于 设计模式 ， 结构型设计模式   |  

享元模式(FlyweightPattern)是一种结构型设计模式。意图：运用共享技术有效的支持大量的细粒度对象。

代码实现

抽象享元角色

Flyweight:享元类的超类，为这些类提供需要的公共接口和抽象类。但是并不强制实现共享。

public abstract class AbstractFlyweight
{
    protected string _color;

    public abstract void Operate();
}

具体享元角色

ConcreteFlyweight:实现享元类共享的接口，还是抽象类。享元类可以在内部实现共享。

public class Flyweight : AbstractFlyweight
{
    public Flyweight(string color)
    {
        _color = color;
    }

    public override void Operate()
    {

        Console.WriteLine(" color: "  + _color);
    }
}

享元工厂

FlyweightFactory:工厂实现角色的创建，并在需要的时候保证共享。

public class Factory
{
    private Dictionary<string, AbstractFlyweight> _dictionary;
    static Factory factory;

    private Factory()
    {
        _dictionary = new Dictionary<string, AbstractFlyweight>();
    }

    public static Factory GetInstance()
    {
        if (null == factory)
        {
            factory = new Factory();
        }
        return factory;
    }

    public AbstractFlyweight Create(string color)
    {
        if ( _dictionary.ContainsKey(color))
        {
            return _dictionary[color];
        }
        else
        {
            AbstractFlyweight flyweight = new Flyweight(color);
            _dictionary.Add(color, flyweight);

            return flyweight;
        }
    }

    public int Count()
    {
        return _dictionary.Count;
    }
}

测试代码

class Program
{
    static void Main(string[] args)
    {
        AbstractFlyweight flyweight1 = Factory.GetInstance().Create("red");
        AbstractFlyweight flyweight2 = Factory.GetInstance().Create("white");
        AbstractFlyweight flyweight3 = Factory.GetInstance().Create("black");
        AbstractFlyweight flyweight4 = Factory.GetInstance().Create("white");
        AbstractFlyweight flyweight5 = Factory.GetInstance().Create("red");

        Console.WriteLine("Instance count is : " + Factory.GetInstance().Count());

        Console.Read();
    }
}

个人理解

享元模式通过共享对象，保证以较少的实例取代多组对象，保证存储和内存的开销。

适用性：

一个应用程序引用了大量的对象。对象的大部分状态可以变成外部状态。如果删除外部状态可以用较少的共享对象代替多组对象。应用程序不依赖对象标示。

优缺点：

享元模式减少了内存和存储的开销。

增加了系统的复杂性。

享元模式将享元对象的状态外部化，读取外部状态的时间变得稍长。

外观模式

发表于 2015-08-25   |   分类于 设计模式 ， 结构型设计模式   |  

外观模式(FacadePattern)是一种结构型设计模式。意图：为子系统中的一组接口提供一个一致的界面，Facade模式定义了一个高层接口，这个接口使得这一子系统更加容易使用。

代码实现

子系统类

提供一系列的具体的操作

public class Tea
{
    public void AddTea()
    {
        Console.WriteLine("加入茶叶");
    }
}

public class Water
{
    public void HeatUpWater()
    {
        Console.WriteLine("烧开水");
    }
}

public class Waiter
{
    public void MakeTea()
    {
        Console.WriteLine("泡茶");
    }
}

外观类

整合子系统的操作，对外提供一个一致的接口，降低客户端和子系统的耦合。抽象的外观类，方便扩展子类外观。

public abstract class AbstractFacade
{
    public abstract void GoodTea();
}

public class Facade : AbstractFacade
{
    private Tea _tea;
    private Water _water;
    private Waiter _waiter;

    public Facade()
    {
        _tea = new Tea();
        _water = new Water();
        _waiter = new Waiter();
    }

    public override void GoodTea()
    {
        _water.HeatUpWater();
        _tea.AddTea();
        _water.HeatUpWater();
    }

}

测试代码

class Program
{
    static void Main(string[] args)
    {
        AbstractFacade facade = new Facade();
        facade.GoodTea();

        Console.Read();
    }
}

个人理解

外观类通过整合一系列的子系统的操作，降低客户端和子系统的耦合。

适用性：

为复杂的子系统提供一个简单的接口。

客户程序和抽象类的实现部分存在一个存在很大的依赖性，引入Facade类可以分离子系统和客户端，使得子系统更加自主、独立。

当你需要构建一个层次结构的子系统的时候，使用facade模式提供子系统每层的入口点。如果子系统相互依赖，可以通过facade在它们之间通讯，从而简化它们的依赖关系。

优缺点：

它屏蔽了子系统空间，减少客户端所有操作的对象的数目，使得客户端的代码变得简单。

它实现了子系统和客户端的松耦合关系，使得子系统的变化的时候不影响客户端，只需要调整facade类即可。

一个子系统的变化不会影响到其他子系统，子系统的内部变化也不会影响到外观对象。

限制了客户端对子系统的访问，减少了可变性和灵活性。

增加子系统可能需要修改外观对象，违背了开闭原则。

模式间的区别：

Facade模式注重简化接口，Adapter模式注重转换接口，Bridge模式注重分离接口（抽象）与其实现，Decorator模式注重稳定接口的前提下为对象扩展功能。

装饰模式

发表于 2015-08-24   |   分类于 设计模式 ， 结构型设计模式   |  

装饰模式(DecoratorPattern)是一种结构型设计模式。意图：动态地给一个对象添加一些额外的职责。就增加功能来说，Decorator 模式相比生成子类更为灵活。

代码实现

抽象构建

compnent:定义了一个接口可以动态的给对象添加职责。

public abstract class AbstractCar
{
    public abstract void Show();
}

具体的构建角色

concreteCompnet:一个实现了compnent接口的对象。这个对象会被装饰，但是它不会访问装饰者的信息，而装饰者不会访问它的实现。

public class ChinaCar : AbstractCar
{
    public override void Show()
    {
        Console.WriteLine("这是一辆国产小汽车");
    }
}

装饰接口

Decorator:包含一个compnent对象的引用。定义了一个与compnent相同的接口。客户端可以处理它们，而不用关心装饰者之间的差别。

public class DecoratorCar : AbstractCar
{
    private AbstractCar _car;
    public DecoratorCar(AbstractCar car)
    {
        this._car = car;
    }

    public override void Show()
    {
        if (null != _car)
        {
            _car.Show();
        }
    }
}

具体的装饰者对象

ConcreteDecorator:它向compnent添加职责，添加额外的装饰功能。

public class LacquerCar : DecoratorCar
{
    public LacquerCar(AbstractCar car) :
        base(car)
    {
    }

    public override void Show()
    {
        LacquerCarRed();
        base.Show();
    }

    private void LacquerCarRed()
    {
        Console.WriteLine("给汽车图绘成红色");
    }
}


public class GpsCar : DecoratorCar
{
    public GpsCar(AbstractCar car) :
        base(car)
    {
    }

    public override void Show()
    {
        LoadGpsForCar();
        base.Show();
    }

    private void LoadGpsForCar()
    {
        Console.WriteLine("给汽车图装上GPS");
    }
}

测试代码

class Program
{
    static void Main(string[] args)
    {
        AbstractCar car = new ChinaCar();
        LacquerCar lacquerCar = new LacquerCar(car);
        lacquerCar.Show();

        GpsCar gpsCar = new GpsCar(lacquerCar);
        gpsCar.Show();

        Console.ReadLine();
    }
}

个人理解

装饰模式避免了构造大量的基类。可以额外的添加大量的功能。与桥接模式的区别：桥接模式主要是在维度上的变化，装饰模式则在一个维度上添加大量额外功能。

适用性：

在不影响其他对象的情况下，用动态、透明的方式给单个对象添加职责。

处理那些可以撤销的职责。

不能采用子类扩展时候。一种是:可能有大量独立的扩展，为了支持每一种扩展，需要产生大量的子类，使子类数目暴涨。另外一种是类定义被隐藏，或者类定义不能用于生成子类。

优缺点：

装饰模式和子类继承都可以扩展对象的功能。但是装饰模式更加灵活。

通过不同的排列组合，可以创造出更多的功能的类。

装饰模式会产生更多的小对象，过度使用，会使程序变得复杂。

与桥接模式的区别：

两者都是为了防止过度继承，子类泛滥的情况。

桥接模式是结构和实现的分离或属性和基于属性行为的分离。装饰模式只是对基于属性的行为封装的类，然后对其进行装饰。例如：异常和异常处理用桥接模式，异常处理添加功能用装饰模式。

桥接模式是横向行为，行为彼此之间没有关联。 装饰模式中的行为具有可叠加性，其表现结果是一个整体，但是其实是各个行为组合后的结果。

组合模式

发表于 2015-08-23   |   分类于 设计模式 ， 结构型设计模式   |  

组合模式（CompsitePattern）是一种结构型设计模式。意图：将对象组合成树形结构以表示“部分-整体”的层次结构，使得用户对单个对象和组合对象的使用具有一致性。

代码实现

组件

public abstract class Compnent
{
    protected string _name;

    public Compnent(string name)
    {
        _name = name;
    }

    public abstract void Add(Compnent compnent);

    public abstract void Remove(Compnent compnent);

    public abstract void Display(int depath);

}

它声明组合中对象的接口。申明获得所有管理子类对象的接口。声明公共接口

组合类

public class Compsite : Compnent
{
    private List<Compnent> _childs = new List<Compnent>();

    public Compsite(string name) : base(name)
    {            
    }

    public override void Add(Compnent compnent)
    {
        _childs.Add(compnent);
    }

    public override void Remove(Compnent compnent)
    {
        _childs.Remove(compnent);
    }

    public override void Display(int depath)
    {
        Console.WriteLine(new string('-', depath) + _name);

        foreach (Compnent compnent in _childs)
        {
            compnent.Display(depath + 2);
        }
    }
}

它实现子类相关的操作。

叶节点

public class Leaf : Compnent
{
    public Leaf(string name)
        : base(name)
    {
    }

    public override void Add(Compnent compnent)
    {            
    }

    public override void Remove(Compnent compnent)
    {
    }

    public override void Display(int depath)
    {
        Console.WriteLine(new string('-', depath) + _name);
    }
}

它实现具体的叶节点的操作。

测试代码

class Program
{
    static void Main(string[] args)
    {
        Compnent rootCompnet = new Compsite("ROOT");

        Compnent rootLeaf1 = new Leaf("ROOTLEAF1");
        rootCompnet.Add(rootLeaf1);

        Compnent rootLeaf2 = new Leaf("ROOTLEAF2");
        rootCompnet.Add(rootLeaf2);

        Compnent rootCompsite= new Compsite("ROOTCOMPSITE");
        Compnent leaf1 = new Leaf("LEAF1");
        rootCompsite.Add(leaf1);
        Compnent leaf2 = new Leaf("LEAF2");
        rootCompsite.Add(leaf2);
        rootCompnet.Add(rootCompsite);

        rootCompnet.Display(0);

        Console.Read();
    }
}

个人理解

组合模式让客户端操作一个树形结构的对象，让客户端的操作统一起来。

适用性：

把对象表示成部分-整体的结构

用户忽略单个对象和组合对象的不同，用户统一的使用组合中的对象。

使得新增类型的组件更加容易。新定义的Composite或Leaf子类自动地与已有的结构和客户代码一起协同工作，客户程序不需因新的Component类而改变。

桥接模式

发表于 2015-08-23   |   分类于 设计模式 ， 结构型设计模式   |  

桥接模式(BridgePattern)是一种结构型设计模式。意图：将抽象部分与它的实现部分分离，使它们都可以独立地变化，从而将抽象与实现解耦。

代码实现

参与者

Abstraction： 1. 定义抽象接口。 2. 拥有一个实现（Implementor）类型对象引用。

RefinedAbstraction: 1. 扩展Abstraction中的接口定义。

public abstract class Shape
{
    protected Corlor _color;

    public abstract void DrawShape();

    public void SetColor(Corlor corlor)
    {
        _color = corlor;
    }
}

public class Rectangle : Shape
{
    public override void DrawShape()
    {
        _color.DrawColor();
        Console.WriteLine("长方形");
    }
}

public class Square : Shape
{
    public override void DrawShape()
    {
        _color.DrawColor();
        Console.WriteLine("正方形");
    } 
}

实现者

Implementor类定义了实现类的接口，这个接口不一定要与Abstraction的接口完全一致；实际上，这两个接口可以完全不同；

ConcreteImplementor类实现了Implementor定义的接口。

public abstract class Corlor
{
    public abstract void DrawColor();
}

public  class Red : Corlor
{
    public override void DrawColor()
    {
        Console.Write("红色的");
    }
}


public  class White : Corlor
{
    public override void DrawColor()
    {
        Console.Write("白色的");
    }
}

测试代码

class Program
{
    static void Main(string[] args)
    {
        Corlor corlor = new Red();
        Shape shape = new Square();

        shape.SetColor(corlor);
        shape.DrawShape();

        Console.Read();
    }
}

个人理解

桥接模式可以用来解决对象有2个或者2个以上的维度变化的问题。对客户隐藏抽象的实现部分（指的是实现者这部分）。

适用性：

你不希望在抽象和它的实现部分有一个固定的绑定关系。

类的抽象和它的实现部分都可以通过子类扩展。

对实现部分的修改，不会影响客户端（主要是指编译）。

如果某个变化在多个对象中共享是，可以抽象该变化，然后该变化的不同因素。

适配器模式

发表于 2015-08-22   |   分类于 设计模式 ， 结构型设计模式   |  

适配器模式(Adapter Pattern)是一种结构型设计模式。意图：将一个类的接口转换成客户希望的另外一个接口。Adapter模式使得原本由于接口不兼容而不能一起工作的那些类可以一起工作。

代码实现

一个接口提供外语的学习方法。原来是讲英语的，现在我们需要它讲法语。同样的接口做不同的工作。提供出来的新的功能的时候，我们可以用适配器模式

类适配器

public interface ITargert
{
    void Request();
}

public class Adaptee
{
    public void SpecialRequest()
    {
        Console.WriteLine("i can speek franch.");
    }
}

public class ClassAdapter : Adaptee, ITargert
{
    public void Request()
    {
        this.SpecialRequest();
    }
}

class Program
{
    static void Main(string[] args)
    {
        ITargert target = new ClassAdapter();
        target.Request();

        Console.Read();
    }
}

通过在类的Request()方法中我们调用Adaptee类的SpecialRequest()来实现法语。

对象适配器

public class Targert : ITargert
{
    public virtual void Request()
    {
        Console.WriteLine("i can speek english.");
    }
}

class ObjectAdapter : Targert
{
    private Adaptee _adaptee;

    public ObjectAdapter()
    {
        _adaptee = new Adaptee();
    }

    public override void Request()
    {
        _adaptee.SpecialRequest();
    }
}

class Program
{
    static void Main(string[] args)
    {
        ITargert targert = new ObjectAdapter();
        targert.Request();

        Console.Read();
    }
}

我们在ObjectAdapter中new一个Adaptee对象出来，然后在方法中Request()调用Adaptee对象的SpecialRequest()，来实现法语。

个人理解

类适配器和对象适配器都很好的复用了现有类。类适配器通过”多继承”的方式，使得它可以从新实现Adaptee的行为，也造成了高耦合的特点。对象适配器通过”对象组合”的方式，使得它松耦合，但是也不能重新定义Adaptee的行为，只能从Adaptee子类中实现。

适用性：

使用一个已经存在的类，而它的接口不符合你的需求。

创建一个可以复用的类，该类可以与其他不相关的类或不可预见的类（即那些接口可能不一定兼容的类）协同工作。

使用一些已经存在的子类，但是不可能对每一个都进行子类化以匹配它们的接口。对象适配器可以适配它的父类接口。

优缺点：

类适配器模式优点：1，可以在不修改原有代码的基础上来复用现有类，很好地符合 “开闭原则”。 2，可以重新定义Adaptee(被适配的类)的部分行为，因为在类适配器模式中，Adapter是Adaptee的子类。 3，仅仅引入一个对象，并不需要额外的字段来引用Adaptee实例（这个即是优点也是缺点）。

类适配器模式缺点：1，用一个具体的Adapter类对Adaptee和Target进行匹配，当如果想要匹配一个类以及所有它的子类时，类的适配器模式就不能胜任了。因为类的适配器模式中没有引入Adaptee的实例，光调用this.SpecificRequest方法并不能去调用它对应子类的SpecificRequest方法。 2， 采用了 “多继承”的实现方式，带来了不良的高耦合。

对象适配器的优点:1, 可以在不修改原有代码的基础上来复用现有类，很好地符合 “开闭原则”（这点是两种实现方式都具有的）。 2，采用 “对象组合”的方式，更符合松耦合。

对象适配器的缺点:1，使得重定义Adaptee的行为较困难，这就需要生成Adaptee的子类并且使得Adapter引用这个子类而不是引用Adaptee本身。
