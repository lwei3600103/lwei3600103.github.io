备忘录模式

发表于 2015-09-18   |   分类于 设计模式 ， 行为型设计模式   |  

备忘录模式(MementoPattern)是一种行为型设计模式。意图：在不破坏封装性的前提下，获取一个对象的内部状态，并在对象外面把它保存下来，这样以后就可以把它恢复到保存的状态。

代码实现

备忘录对象

备忘录对象(Memento)存储对象的内部状态，在需要的时候提供发起人需要的内部状态。

class Memento
{
    private string _state;

    public void SetState(string state)
    {
        _state = state;
    }

    public string State()
    {
        return _state;
    }
}

发起人对象

发起人对象（Originator）记录当前时刻的内部状态，负责定义那些属于备份范围的状态，创建和恢复备忘数据。

class Originator
{
    private string _state;

    public string State
    {
        get { return _state; }
        set
        {
            _state = value;
            Console.WriteLine("Current state: " + _state);
        }
    }

    public Memento CreateMemento()
    {
        Memento memento = new Memento();
        memento.SetState(_state);


        return memento;
    }

    public void RestoreState(Memento memento)
    {
        Console.WriteLine("start to restore state.");
        this.State = memento.State();
        Console.WriteLine("Current state: " + _state);
    }
}

管理角色

管理角色（Caretaker）对备忘录管理，保存和提供备忘录。

class Caretaker
{
    private Memento _memento;

    public void SetMemento(Memento memento)
    {
        _memento = memento;
    }

    public Memento GetMemento()
    {
        return _memento;
    }
}

测试代码

class Program
{
    static void Main(string[] args)
    {
        Originator originator = new Originator();
        originator.State = "state1";
        Caretaker caretaker = new Caretaker();
        caretaker.SetMemento(originator.CreateMemento());

        originator.State = "state2";
        originator.RestoreState(caretaker.GetMemento());

        Console.Read();

    }
}

个人理解

备忘录模式在不破坏内部结构和整体继承关系的前提下，保存内部状态，恢复数据。

适用性

必须保存一个对象在某一时刻的状态，这样在以后就可以恢复到当前这一时刻的状态。如果用接口来保存这一状态，就会暴露内部状态出去，破坏封装性。

发起人状态改变的时候，可能是错误的改变，可以恢复到原来的状态。

备份的状态保存在发起人外部，而不是内部。

备忘录模式的缺点保存过多的状态，对资源消耗比较大。

中介者模式

发表于 2015-09-16   |   分类于 设计模式 ， 行为型设计模式   |  

中介者模式(MediatorPattern)是一种行为型设计模式。意图：用一个中介对象封装一系列的对象的交互。中介者使各个对象不需要的显示引用，从而使其耦合松散，并且可以独立的改变他们之间的交互。

代码实现

抽象同事类

抽象同事类（Colleague）主要负责约束同事类的类型，并实现一些同事类的公共功能。

abstract class Colleague
{
    protected AbstractMediator _mediator;
    public virtual void SetMediator(Mediator mediator)
    {
        _mediator = mediator;
    }

    public virtual void SendMessage(string message)
    {
        if (null != _mediator)
        {
            _mediator.SendMessage(message, this);
        }
    }

    public abstract void ReceiveMessage(string message);
}

同事类

同事类实现自己的业务，当与中介者有交互的时候就通知中介者。

class Button : Colleague
{
    public override void ReceiveMessage(string message)
    {
        Console.WriteLine("button receive " + message);
    }
}

class TextEdit : Colleague
{
    public override void ReceiveMessage(string message)
    {
        Console.WriteLine("TextEdit receive " + message);
        SendMessage(" TextEdit has receive message. ");
    }
}

抽象中介者类

抽象中介者定义各个同事间需要的方法，可以是公共方法。

abstract class AbstractMediator
{
    protected IList<object> _list;

    public abstract void SendMessage(string message, object obj);

    public abstract void AddColleague(object obj);

}

中介者类

中介者类了解各个同事对象，并协调各个同事对象的交互。

class Mediator : AbstractMediator
{
    public override void AddColleague(object obj)
    {
        if (null == _list)
        {
            _list = new List<object>();
        }

        _list.Add(obj);
    }

    public override void SendMessage(string message, object obj)
    {
        foreach (Colleague curObject in _list)
        {
            if (curObject.GetType().Name != obj.GetType().Name)
            {
                curObject.ReceiveMessage(message);
            }
        }
    }

}

测试代码

class Program
{
    static void Main(string[] args)
    {
        Mediator mediator = new Mediator();
        Button button = new Button();
        button.SetMediator(mediator);

        TextEdit edit1 = new TextEdit();
        edit1.SetMediator(mediator);

        TextEdit edit2 = new TextEdit();
        edit2.SetMediator(mediator);

        mediator.AddColleague(button);
        mediator.AddColleague(edit1);
        mediator.AddColleague(edit2);

        button.SendMessage("button clicked. ");

        Console.Read();

    }
}

个人理解

中介者封装各个对象的交互，使得原来的网状接口变成星状结构。

适用性

一组对象以定义良好但是复杂的方式进行通信。产生的相互依赖关系结构混乱且难以理解。

一个对象引用其他很多对象，并且直接和这些对象通信，导致难以复用这些对象。

想定义一个分布在多个类的行为，又不想生成多个子类。

迭代器模式

发表于 2015-09-06   |   分类于 设计模式 ， 行为型设计模式   |  

迭代器模式(IteratorPattern)是一种行为型设计模式。意图：提供一种方法顺序访问一个聚合对象中的各个元素，而不暴露该对象的内部表示。

代码实现

抽象迭代器角色

抽象迭代器角色定义访问和遍历聚合对象的接口和方法。

abstract class AbstractIterator
{
    public abstract object First();
    public abstract object Next();
    public abstract object CurrentItem();
    public abstract bool IsDone();
}

迭代器角色

迭代器角色实现抽象迭代器的接口和方法，并记录当前访问的位置。

class ConcreteIterator : AbstractIterator
{
    private Aggregate _aggregate;
    private int _current = 0;

    public ConcreteIterator(Aggregate aggregate)
    {
        _aggregate = aggregate;
    }

    public override object First()
    {
        return _aggregate[0];
    }

    public override object Next()
    {
        if (_current < _aggregate.Count)
        {
            return _aggregate[_current++];
        }

        return null;
    }

    public override object CurrentItem()
    {
        if ( _current < _aggregate.Count)
        {
            return _aggregate[_current];
        }

        return null;
    }

    public override bool IsDone()
    {
        return _current >= _aggregate.Count ? true : false;
    }
}

抽象集合角色

抽象集合角色提供创建集合角色的接口。

abstract class AbstractAggregate
{
    public abstract AbstractIterator CreateIterator(); 
}

集合角色

集合角色实现创建集合的接口。并提供遍历和访问集合的相关的接口或方法给迭代器。

class Aggregate : AbstractAggregate
{
    private IList<object> items = new List<object>();

    public override AbstractIterator CreateIterator()
    {
        return new ConcreteIterator(this);
    }

    public int Count
    {
        get { return items.Count; }
    }

    public object this[int index]
    {
        get {return items[index];}
        set { items.Insert(index, value); }
    }
}

测试代码

struct Test
{
    public string code;
    public string name;
}

class Program
{
    static void Main(string[] args)
    {
        Aggregate aggregate = new Aggregate();
        aggregate[0] = 1;
        aggregate[1] = "张三在上街";
        aggregate[2] = 1.6;

        Test test = new Test();
        test.code = "001";
        test.name = "asf";
        aggregate[3] = test;

        AbstractIterator iterator = new ConcreteIterator(aggregate);
        object first = iterator.First();
        while (!iterator.IsDone())
        {
            object test1 = iterator.CurrentItem();
            Console.WriteLine(iterator.CurrentItem().ToString());
            iterator.Next();
        }

        Console.Read();
    }
}

个人理解

迭代器模式提供顺序遍历集合的方法。

适用性

访问一个聚合对象而不暴露它内部的实现。

支持对聚合对象的多重遍历。

为遍历不同的聚合结构提供统一的接口。

命令行模式

发表于 2015-09-05   |   分类于 设计模式 ， 行为型设计模式   |  

命令行模式(CommandPattern)是一种行为型设计模式。意图：将一个请求封装成一个对象，从而使你可用不同的请求对客户进行参数化。对请求排队、记录日志以及可撤销的操作等。

代码实现1

接收者

Receiver接收调用者请求的实际去操作的对象。

class ReceiverTV
{

    private int _currentChannel;

    public ReceiverTV()
    {
        _currentChannel = 0;
    }

    public void TurnOn()
    {
        Console.WriteLine("打开电视机。");
    }

    public void TurnOff()
    {
        Console.WriteLine("关闭电视机");
    }

    public void ChangeChannel()
    {
        Console.WriteLine("调整频道为：" + (++_currentChannel));
    }
}

抽象命令对象

ICommand用于解耦目标对象与调用者之间的抽象命令对象。

interface ICommand
{
   void Execute();
}

命令实现的对象

ConcreteCommand是ICommand的实现类，实现ICommand中的接口。

class TurnOn : ICommand
{
    private ReceiverTV _tv;

    public TurnOn(ReceiverTV tv)
    {
        _tv = tv;
    }

    public void Execute()
    {
        _tv.TurnOn();
    }
}

class TurnOff : ICommand
{
    private ReceiverTV _tv;

    public TurnOff(ReceiverTV tv)
    {
        _tv = tv;
    }

    public void Execute()
    {
        _tv.TurnOff();
    }
}

class ChangeChannel : ICommand
{
    private ReceiverTV _tv;

    public ChangeChannel(ReceiverTV tv)
    {
        _tv = tv;
    }

    public void Execute()
    {
        _tv.ChangeChannel();
    }
}

调用者

Invoker调用者负责调用对象。

class Invoker
{
    private List<ICommand> _list;

    public void Add(ICommand command)
    {
        if (null == _list)
        {
            _list = new List<ICommand>();
        }

        _list.Add(command);
    }

    public void Run()
    {
        foreach (ICommand command in _list)
        {
            command.Execute();
        }
    }
}

测试代码

class Program
{
    static void Main(string[] args)
    {
        ReceiverTV tv = new ReceiverTV();

        Invoker invoker = new Invoker();

        ICommand commandTurnOn = new TurnOn(tv);
        invoker.Add(commandTurnOn);

        ICommand commandChangerChannel = new ChangeChannel(tv);
        invoker.Add(commandChangerChannel);

        ICommand commandTurnOff = new TurnOff(tv);
        invoker.Add(commandTurnOff);

        invoker.Run();


        Console.Read();
    }
}

调用者

public delegate void HandleExecute();

class EventHandleInvoker

        _list.Add(handle);

代码实现3（异步命令）

异步接口命令

interface IAsynCommandBase : ICommand
{
    event AsyncCallback _asynCallback;

    event EventHandler _handler;

    void AsynExecute();
}

异步命令抽象实现

class AsynCommandBase : IAsynCommandBase
{
    public event AsyncCallback _asynCallback;

    public event EventHandler _handler;

    public void AsynExecute()
    {
        if (null != _handler && null != _asynCallback)
        {
            _handler.BeginInvoke(this, new EventArgs(), _asynCallback, null);
        }
    }

    public void Execute()
    {
    }
}

实体内容

class TestCommand : AsynCommandBase
{
    public TestCommand()
    {
        _asynCallback += new AsyncCallback(TestCommand__asynCallback);
        _handler += new EventHandler(TestCommand__handler);
    }

    public void TestCommand__asynCallback(IAsyncResult asynBase)
    {
        Console.WriteLine("TestCommand__asynCallback");
    }

    public void TestCommand__handler(Object sender, EventArgs args)
    {
        Console.WriteLine("TestCommand__handler");
    }
}

测试代码

class Program
{
    static void Main(string[] args)
    {
        IAsynCommandBase asyn = new TestCommand();
        asyn.AsynExecute();
        System.Threading.Thread.Sleep(10000);

        Console.Read();
    }
}

个人理解

命令模式解耦了调用者和接受者间关系。调用者只需要知道命令即可，不用管实际接受者的具体操作。

适用性

使用命令模式作为回调函数（callBack）的替代。

异步执行的请求或操作。即使请求者已经销毁了，调用者依旧执行操作。

系统支持撤销、恢复等操作。通过记录的日志重新执行命令Execute或Undo。

责任链模式

发表于 2015-09-04   |   分类于 设计模式 ， 行为型设计模式   |  

责任链模式(ChainOfResponsibilityPattern)是一种行为型设计模式。意图：使多个对象都有机会处理请求，从而避免发送者和接收者之间的耦合关系。将所有可以处理请求的对象串成一条链，并沿着这条链传递请求，直到有对象处理它。

代码实现

请求对象

class Request
{
    private int _day;
    private string _reason;


    public int Day
    {
        get { return _day; }
        set { _day = value; }
    }

    public string Reason 
    {
        get { return _reason; }
        set { _reason = value; }
    }
}

一个抽象“链”对象

抽象对象定义抽象的处理请求的方法，并定义下个链对象。

abstract class AbstractHandler
{
    private AbstractHandler _successor;
    protected string _name;

    public AbstractHandler(string name)
    {
        _name = name;
    }

    public AbstractHandler Successor 
    {
        get { return _successor; }
        set { _successor = value; }
    }

    public abstract bool PassHandle(Request request);
}

具体的请求处理对象

具体的对象根据条件实现请求的方法。

class PM : AbstractHandler
{
    public PM(string name) :
        base(name)
    {

    }

    public override bool PassHandle(Request request)
    {
        if (request.Day < 0.5)
        {
            Console.WriteLine("PM " + _name + " 批准请假。 ");
            return true;
        }

        return Successor.PassHandle(request);
    }
}

class HR : AbstractHandler
{
    public HR(string name) :
        base(name)
    {

    }

    public override bool PassHandle(Request request)
    {
        if (request.Day > 0.5 && request.Day < 3) 
        {
            Console.WriteLine("HR " + _name + " 批准请假。 ");
            return true;
        }

        return Successor.PassHandle(request);
    }
}

class Manager : AbstractHandler
{
    public Manager(string name) :
        base(name)
    {

    }

    public override bool PassHandle(Request request)
    {
        if (request.Reason.Equals("正当理由"))
        {
            Console.WriteLine("Manager " + _name + " 批准请假。 ");
            return true;
        }

        Console.WriteLine("Manager " + _name + " 认为理由不正当，不允许批准。 ");
        return false;
    }
}

测试代码

class Program
{
    static void Main(string[] args)
    {
        PM pm = new PM("项目经理");
        HR hr = new HR("人力资源");
        Manager manager = new Manager("总经理");

        pm.Successor = hr;
        hr.Successor = manager;


        Request request1 = new Request();
        request1.Day = 2;
        request1.Reason = "正当理由";

        pm.PassHandle(request1);

        Request request2 = new Request();
        request2.Day = 6;
        request2.Reason = "不正当理由";

        pm.PassHandle(request2);

        Console.Read();
    }
}

个人理解

责任链模式通过一条链来处理请求，请求者并不知道处理者对象，分离了请求者和处理者。但是每个请求都必须从链头开始处理，造成了内存中的垃圾对象。

适用性

有多个对象处理请求时，而且请求在运行时候决定。

在不明确处理者的时候，向多个对象提出一个请求。

可处理一个请求的对象集合应该被动态指定。

如果存在N对N，或者是常规线性关系，我们完全可以用表驱动代替责任链。

责任链并非只可以在上下级间传递，也可以在同级间传递。

模板模式

发表于 2015-09-03   |   分类于 设计模式 ， 行为型设计模式   |  

模板模式(TemplatePattern)是一种行为型设计模式。意图：定义一个算法的框架，而将其中的一些步骤的实现延迟到子类中。

代码实现

抽象模板类

抽象模板类定义算法的框架，定义抽象行为，让其在子类中实现。

abstract class AbstractBeverage
{
    public void Create()
    {
        BoilWater();
        AddCoundiments();
        Brew();
        PourInCup();

        if (Hook())
        {
            Again();
        }
    }

    public void BoilWater()
    {
        Console.WriteLine("烧开水");
    }

    public abstract void AddCoundiments();

    public abstract void Brew();

    public void PourInCup()
    {
        Console.WriteLine("倒入杯子中");
    }

    protected virtual bool Hook()
    {
        return false;
    }

    private void Again()
    {
        Console.WriteLine("在来一杯");
    }
}

具体实现的子类

子类实现抽象函数中的方法，实现个性化。

class Tea : AbstractBeverage
{
    public override void AddCoundiments()
    {
        Console.WriteLine("加入茶叶");
    }

    public override void Brew()
    {
        Console.WriteLine("冲泡茶叶");
    }


    protected override bool Hook()
    {
        return true;
    }
}

class Coffer : AbstractBeverage
{
    public override void AddCoundiments()
    {
        Console.WriteLine("加入咖啡豆");
    }

    public override void Brew()
    {
        Console.WriteLine("冲泡咖啡");
        Season();
    }

    private void Season()
    {
        Console.WriteLine("加入2块方糖，调节味道");
    }
}

测试代码

class Program
{
    static void Main(string[] args)
    {
        AbstractBeverage beverageTea = new Tea();
        beverageTea.Create();

        AbstractBeverage beverageCoffer = new Coffer();
        beverageCoffer.Create();

        Console.Read();
    }
}

个人理解

模板方法通过定义一个相同的算法框架，而将其中部分步骤的实现延迟到子类中实现。

适用性

一次性实现算法的固定部分，将可变的行为留给子类。

各个子类中的公有行为被提取到超类中，提高了代码的可复用性。

通过钩子（hook）控制了子类的扩展。子类可以实现也可以不是先钩子方法。

解释器模式

发表于 2015-09-02   |   分类于 设计模式 ， 行为型设计模式   |  

解释器模式(InterpreterPattern)是一种行为型设计模式。意图：给定一个语言，定义它的文法的一种表示，并定义一个解释器，这个解释器使用该表示来解释语言中的句子。

代码实现

抽象解释器

抽象解释器声明一个所有具体的表达式所要实现的接口。

abstract class Expression
{
    public abstract int Interpreter(Dictionary<string, int> var);
}
终结符表达式

终结符表达式实现与文法中相关联的解释操作。

class VarExpression : Expression
{
    private string _key;
    public VarExpression(string key)
    {
        _key = key;
    }

    public override int Interpreter(Dictionary<string, int> var)
    {
        return var[_key];
    }
}
非终结符表达式

文法中每条规则对应一个非终结符表达式。非终结符表达式一般是指“+”等这种。

abstract class SymbolExpression : Expression
{
    protected Expression _left;
    protected Expression _right;

    public SymbolExpression(Expression left, Expression right)
    {
        _left = left;
        _right = right;
    }
}


class AddExpression : SymbolExpression
{
    public AddExpression(Expression left, Expression right) :
        base(left, right)
    {
    }

    public override int Interpreter(Dictionary<string, int> var)
    {
        return base._left.Interpreter(var) + base._right.Interpreter(var);
    }
}

class SubExpression : SymbolExpression
{
    public SubExpression(Expression left, Expression right) :
        base(left, right)
    {
    }

    public override int Interpreter(Dictionary<string, int> var)
    {
        return base._left.Interpreter(var) - base._right.Interpreter(var);
    }
}
封装的解释器
class Calculator
{
    private Expression _expression;
    private Dictionary<string, int> _dictionary;
    public Calculator(string str)
    {
        string[] listStr = str.Split(new Char[] { ' ' }, StringSplitOptions.RemoveEmptyEntries);
        GetDictonary(listStr);

        Expression left = null;
        Expression right = null;
        Expression expression = null;

        for (int i = 0; i < listStr.Length; i++)
        {
            switch (listStr[i])
            {
                case "+":
                    left = expression;
                    right = new VarExpression(listStr[++i]);
                    expression = new AddExpression(left, right);
                    break;
                case "-":
                    left = expression;
                    right = new VarExpression(listStr[++i]);
                    expression = new SubExpression(left, right);
                    break;
                default:
                    expression = new VarExpression(listStr[i]);
                    break;
            }
        }

        _expression = expression;
    }

    public int Run()
    {
        return this._expression.Interpreter(_dictionary);
    }


    private void GetDictonary(string[] listStr )
    {
        _dictionary = new Dictionary<string, int>();

        foreach (string curString in listStr)
        {
            if (curString != "+" && curString != "-")
            {
                _dictionary.Add(curString, Convert.ToInt32(curString));
            }
        }
    }
}
测试代码
class Program
{
    static void Main(string[] args)
    {
        string str = "1234 + 234234 - 948 + 3242 - 4994 + 3";
        Calculator calculator = new Calculator(str);
        Console.WriteLine(calculator.Run());

        Console.Read();
    }

}

个人理解

解释器是一个语法分析工具。

适用性

当一个语言需要解释执行，你可以将语言中的句子表示为抽象的语法树时。

有一个简单的语法规则（如：sql）,一些重复的操作（如：+ - * /）。

优缺点

增加一个规则只需要增加一个非终结符表达式。修改一个规则只需要修改非终结符表达式。具有良好的扩展性。

解释器模式会引起类的膨胀。每个语法都会产生一个非终结符表达式。而且进行语法解释的时候，会使用回调，影响效率。

代理模式

发表于 2015-08-30   |   分类于 设计模式 ， 创建型设计模式   |  

代理模式(ProxyPattern)是一种创建型设计模式。意图：为其他对象提供一种代理以控制对这个对象的访问。

代码实现

普通代理

接口类

提供公有的方法

public interface IGamePlayer
{
    void Login(string user, string password);

    void KillBoss();

    void Update();

    IGamePlayer GetProxy();
}
实例类

完成接口的方法

public class GamePlayer : IGamePlayer
{
    private string _user;

    public void Login(string user, string password)
    {
        _user = user;
        Console.WriteLine("user is " + user + " password is " + password);
    }

    public void KillBoss()
    {
        Console.WriteLine(_user + " kill a boss.");
    }

    public void Update()
    {
        Console.WriteLine(_user + " update one. ");
    }

    public IGamePlayer GetProxy()
    {
        return null;
    }
}
代理类
public class Proxy : IGamePlayer
{

    {

        _player.KillBoss();

    }
}

强制代理

一般都是通过代理找到真实对象，强制代理却通过真实对象找到代理。

实体类

强制代理 只需要将实例类中的方法加入获得代理。 代理必须是真实对象，不能是外部new出来的。

public class ForceGamePlayer : IGamePlayer
{
    private string _user;
    private IGamePlayer _proxy = null;

    public IGamePlayer GetProxy()
    {
        _proxy = new Proxy(this);
        return _proxy;
    }

    public void Login(string user, string password)
    {
        _user = user;

        if (IsProxy())
        {
            Console.WriteLine("user is " + user + " password is " + password);
        }
        else
        {
            Console.WriteLine(" please use proxy. ");
        }

    }

    public void KillBoss()
    {
        if (IsProxy())
        {
            Console.WriteLine(_user + " kill a boss.");
        }
        else
        {
            Console.WriteLine(" please use proxy. ");
        }

    }

    public void Update()
    {
        if (IsProxy())
        {
            Console.WriteLine(_user + " update one. ");
        }
        else
        {
            Console.WriteLine(" please use proxy. ");
        }

    }

    public bool IsProxy()
    {
        if (null == _proxy)
        {
            return false;
        }
        else
        {
            return true;
        }
    }
}

虚拟代理

在需要的时候才判断是否创建对象，可以避免代理对象较多引起的初始化缓慢的问题，缺点是要在每个方法中判断时候对象是否为null。

公用接口类
public interface Subject
{
    void Request();
}
实例类
public class RealSubject : Subject
{
    public void Request()
    {
        Console.WriteLine(" real subject ");
    }
}
虚拟代理类
public class VirtualProxy : Subject
{
    private Subject _subject;

    public void Request()
    {
        if (null == _subject)
        {
            _subject = new RealSubject();
        }

        _subject.Request();
    }
}
测试代码
class Program
{
    static void Main(string[] args)
    {
        VirtualProxy virtualProxy = new VirtualProxy();
        virtualProxy.Request();

        Console.Read();
    }
}

个人理解

在代理模式中，真实的角色实现业务逻辑，而不关心事务，事务由代理对象来处理。真实对象，只要实现了接口，不管怎么变化，代理类都不需要变化。

适用性

在要用比较通用和复杂的指针代替简单的指针的时候。

proxy并不一定要求保持接口的一致性，只要能够实现间接控制，有时候损及一些透明性是可以接受的。

“增加一层间接层”是软件系统中对许多负责问题的一种常见解决方法。

适配器Adapter 为它所适配的对象提供了一个不同的接口。相反，代理提供了与它的实体相同的接口。

尽管Decorator的实现部分与代理相似，但Decorator的目的不一样。Decorator为对象添加一个或多个功能，而代理则控制对对象的访问。

享元模式

发表于 2015-08-26   |   分类于 设计模式 ， 结构型设计模式   |  

享元模式(FlyweightPattern)是一种结构型设计模式。意图：运用共享技术有效的支持大量的细粒度对象。

代码实现

抽象享元角色

Flyweight:享元类的超类，为这些类提供需要的公共接口和抽象类。但是并不强制实现共享。

public abstract class AbstractFlyweight
{
    protected string _color;

    public abstract void Operate();
}

具体享元角色

ConcreteFlyweight:实现享元类共享的接口，还是抽象类。享元类可以在内部实现共享。

public class Flyweight : AbstractFlyweight
{
    public Flyweight(string color)
    {
        _color = color;
    }

    public override void Operate()
    {

        Console.WriteLine(" color: "  + _color);
    }
}

享元工厂

FlyweightFactory:工厂实现角色的创建，并在需要的时候保证共享。

public class Factory
{
    private Dictionary<string, AbstractFlyweight> _dictionary;
    static Factory factory;

    private Factory()
    {
        _dictionary = new Dictionary<string, AbstractFlyweight>();
    }

    public static Factory GetInstance()
    {
        if (null == factory)
        {
            factory = new Factory();
        }
        return factory;
    }

    public AbstractFlyweight Create(string color)
    {
        if ( _dictionary.ContainsKey(color))
        {
            return _dictionary[color];
        }
        else
        {
            AbstractFlyweight flyweight = new Flyweight(color);
            _dictionary.Add(color, flyweight);

            return flyweight;
        }
    }

    public int Count()
    {
        return _dictionary.Count;
    }
}

测试代码

class Program
{
    static void Main(string[] args)
    {
        AbstractFlyweight flyweight1 = Factory.GetInstance().Create("red");
        AbstractFlyweight flyweight2 = Factory.GetInstance().Create("white");
        AbstractFlyweight flyweight3 = Factory.GetInstance().Create("black");
        AbstractFlyweight flyweight4 = Factory.GetInstance().Create("white");
        AbstractFlyweight flyweight5 = Factory.GetInstance().Create("red");

        Console.WriteLine("Instance count is : " + Factory.GetInstance().Count());

        Console.Read();
    }
}

个人理解

享元模式通过共享对象，保证以较少的实例取代多组对象，保证存储和内存的开销。

适用性：

一个应用程序引用了大量的对象。对象的大部分状态可以变成外部状态。如果删除外部状态可以用较少的共享对象代替多组对象。应用程序不依赖对象标示。

优缺点：

享元模式减少了内存和存储的开销。

增加了系统的复杂性。

享元模式将享元对象的状态外部化，读取外部状态的时间变得稍长。

外观模式

发表于 2015-08-25   |   分类于 设计模式 ， 结构型设计模式   |  

外观模式(FacadePattern)是一种结构型设计模式。意图：为子系统中的一组接口提供一个一致的界面，Facade模式定义了一个高层接口，这个接口使得这一子系统更加容易使用。

代码实现

子系统类

提供一系列的具体的操作

public class Tea
{
    public void AddTea()
    {
        Console.WriteLine("加入茶叶");
    }
}

public class Water
{
    public void HeatUpWater()
    {
        Console.WriteLine("烧开水");
    }
}

public class Waiter
{
    public void MakeTea()
    {
        Console.WriteLine("泡茶");
    }
}

外观类

整合子系统的操作，对外提供一个一致的接口，降低客户端和子系统的耦合。抽象的外观类，方便扩展子类外观。

public abstract class AbstractFacade
{
    public abstract void GoodTea();
}

public class Facade : AbstractFacade
{
    private Tea _tea;
    private Water _water;
    private Waiter _waiter;

    public Facade()
    {
        _tea = new Tea();
        _water = new Water();
        _waiter = new Waiter();
    }

    public override void GoodTea()
    {
        _water.HeatUpWater();
        _tea.AddTea();
        _water.HeatUpWater();
    }

}

测试代码

class Program
{
    static void Main(string[] args)
    {
        AbstractFacade facade = new Facade();
        facade.GoodTea();

        Console.Read();
    }
}

个人理解

外观类通过整合一系列的子系统的操作，降低客户端和子系统的耦合。

适用性：

为复杂的子系统提供一个简单的接口。

客户程序和抽象类的实现部分存在一个存在很大的依赖性，引入Facade类可以分离子系统和客户端，使得子系统更加自主、独立。

当你需要构建一个层次结构的子系统的时候，使用facade模式提供子系统每层的入口点。如果子系统相互依赖，可以通过facade在它们之间通讯，从而简化它们的依赖关系。

优缺点：

它屏蔽了子系统空间，减少客户端所有操作的对象的数目，使得客户端的代码变得简单。

它实现了子系统和客户端的松耦合关系，使得子系统的变化的时候不影响客户端，只需要调整facade类即可。

一个子系统的变化不会影响到其他子系统，子系统的内部变化也不会影响到外观对象。

限制了客户端对子系统的访问，减少了可变性和灵活性。

增加子系统可能需要修改外观对象，违背了开闭原则。

模式间的区别：

Facade模式注重简化接口，Adapter模式注重转换接口，Bridge模式注重分离接口（抽象）与其实现，Decorator模式注重稳定接口的前提下为对象扩展功能。
