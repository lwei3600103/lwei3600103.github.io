外观模式

发表于 2015-08-25   |   分类于 设计模式 ， 结构型设计模式   |  

外观模式(FacadePattern)是一种结构型设计模式。意图：为子系统中的一组接口提供一个一致的界面，Facade模式定义了一个高层接口，这个接口使得这一子系统更加容易使用。

代码实现

子系统类

提供一系列的具体的操作

public class Tea
{
    public void AddTea()
    {
        Console.WriteLine("加入茶叶");
    }
}

public class Water
{
    public void HeatUpWater()
    {
        Console.WriteLine("烧开水");
    }
}

public class Waiter
{
    public void MakeTea()
    {
        Console.WriteLine("泡茶");
    }
}

外观类

整合子系统的操作，对外提供一个一致的接口，降低客户端和子系统的耦合。抽象的外观类，方便扩展子类外观。

public abstract class AbstractFacade
{
    public abstract void GoodTea();
}

public class Facade : AbstractFacade
{
    private Tea _tea;
    private Water _water;
    private Waiter _waiter;

    public Facade()
    {
        _tea = new Tea();
        _water = new Water();
        _waiter = new Waiter();
    }

    public override void GoodTea()
    {
        _water.HeatUpWater();
        _tea.AddTea();
        _water.HeatUpWater();
    }

}

测试代码

class Program
{
    static void Main(string[] args)
    {
        AbstractFacade facade = new Facade();
        facade.GoodTea();

        Console.Read();
    }
}

个人理解

外观类通过整合一系列的子系统的操作，降低客户端和子系统的耦合。

适用性：

为复杂的子系统提供一个简单的接口。

客户程序和抽象类的实现部分存在一个存在很大的依赖性，引入Facade类可以分离子系统和客户端，使得子系统更加自主、独立。

当你需要构建一个层次结构的子系统的时候，使用facade模式提供子系统每层的入口点。如果子系统相互依赖，可以通过facade在它们之间通讯，从而简化它们的依赖关系。

优缺点：

它屏蔽了子系统空间，减少客户端所有操作的对象的数目，使得客户端的代码变得简单。

它实现了子系统和客户端的松耦合关系，使得子系统的变化的时候不影响客户端，只需要调整facade类即可。

一个子系统的变化不会影响到其他子系统，子系统的内部变化也不会影响到外观对象。

限制了客户端对子系统的访问，减少了可变性和灵活性。

增加子系统可能需要修改外观对象，违背了开闭原则。

模式间的区别：

Facade模式注重简化接口，Adapter模式注重转换接口，Bridge模式注重分离接口（抽象）与其实现，Decorator模式注重稳定接口的前提下为对象扩展功能。

装饰模式

发表于 2015-08-24   |   分类于 设计模式 ， 结构型设计模式   |  

装饰模式(DecoratorPattern)是一种结构型设计模式。意图：动态地给一个对象添加一些额外的职责。就增加功能来说，Decorator 模式相比生成子类更为灵活。

代码实现

抽象构建

compnent:定义了一个接口可以动态的给对象添加职责。

public abstract class AbstractCar
{
    public abstract void Show();
}

具体的构建角色

concreteCompnet:一个实现了compnent接口的对象。这个对象会被装饰，但是它不会访问装饰者的信息，而装饰者不会访问它的实现。

public class ChinaCar : AbstractCar
{
    public override void Show()
    {
        Console.WriteLine("这是一辆国产小汽车");
    }
}

装饰接口

Decorator:包含一个compnent对象的引用。定义了一个与compnent相同的接口。客户端可以处理它们，而不用关心装饰者之间的差别。

public class DecoratorCar : AbstractCar
{
    private AbstractCar _car;
    public DecoratorCar(AbstractCar car)
    {
        this._car = car;
    }

    public override void Show()
    {
        if (null != _car)
        {
            _car.Show();
        }
    }
}

具体的装饰者对象

ConcreteDecorator:它向compnent添加职责，添加额外的装饰功能。

public class LacquerCar : DecoratorCar
{
    public LacquerCar(AbstractCar car) :
        base(car)
    {
    }

    public override void Show()
    {
        LacquerCarRed();
        base.Show();
    }

    private void LacquerCarRed()
    {
        Console.WriteLine("给汽车图绘成红色");
    }
}


public class GpsCar : DecoratorCar
{
    public GpsCar(AbstractCar car) :
        base(car)
    {
    }

    public override void Show()
    {
        LoadGpsForCar();
        base.Show();
    }

    private void LoadGpsForCar()
    {
        Console.WriteLine("给汽车图装上GPS");
    }
}

测试代码

class Program
{
    static void Main(string[] args)
    {
        AbstractCar car = new ChinaCar();
        LacquerCar lacquerCar = new LacquerCar(car);
        lacquerCar.Show();

        GpsCar gpsCar = new GpsCar(lacquerCar);
        gpsCar.Show();

        Console.ReadLine();
    }
}

个人理解

装饰模式避免了构造大量的基类。可以额外的添加大量的功能。与桥接模式的区别：桥接模式主要是在维度上的变化，装饰模式则在一个维度上添加大量额外功能。

适用性：

在不影响其他对象的情况下，用动态、透明的方式给单个对象添加职责。

处理那些可以撤销的职责。

不能采用子类扩展时候。一种是:可能有大量独立的扩展，为了支持每一种扩展，需要产生大量的子类，使子类数目暴涨。另外一种是类定义被隐藏，或者类定义不能用于生成子类。

优缺点：

装饰模式和子类继承都可以扩展对象的功能。但是装饰模式更加灵活。

通过不同的排列组合，可以创造出更多的功能的类。

装饰模式会产生更多的小对象，过度使用，会使程序变得复杂。

与桥接模式的区别：

两者都是为了防止过度继承，子类泛滥的情况。

桥接模式是结构和实现的分离或属性和基于属性行为的分离。装饰模式只是对基于属性的行为封装的类，然后对其进行装饰。例如：异常和异常处理用桥接模式，异常处理添加功能用装饰模式。

桥接模式是横向行为，行为彼此之间没有关联。 装饰模式中的行为具有可叠加性，其表现结果是一个整体，但是其实是各个行为组合后的结果。

组合模式

发表于 2015-08-23   |   分类于 设计模式 ， 结构型设计模式   |  

组合模式（CompsitePattern）是一种结构型设计模式。意图：将对象组合成树形结构以表示“部分-整体”的层次结构，使得用户对单个对象和组合对象的使用具有一致性。

代码实现

组件

public abstract class Compnent
{
    protected string _name;

    public Compnent(string name)
    {
        _name = name;
    }

    public abstract void Add(Compnent compnent);

    public abstract void Remove(Compnent compnent);

    public abstract void Display(int depath);

}

它声明组合中对象的接口。申明获得所有管理子类对象的接口。声明公共接口

组合类

public class Compsite : Compnent
{
    private List<Compnent> _childs = new List<Compnent>();

    public Compsite(string name) : base(name)
    {            
    }

    public override void Add(Compnent compnent)
    {
        _childs.Add(compnent);
    }

    public override void Remove(Compnent compnent)
    {
        _childs.Remove(compnent);
    }

    public override void Display(int depath)
    {
        Console.WriteLine(new string('-', depath) + _name);

        foreach (Compnent compnent in _childs)
        {
            compnent.Display(depath + 2);
        }
    }
}

它实现子类相关的操作。

叶节点

public class Leaf : Compnent
{
    public Leaf(string name)
        : base(name)
    {
    }

    public override void Add(Compnent compnent)
    {            
    }

    public override void Remove(Compnent compnent)
    {
    }

    public override void Display(int depath)
    {
        Console.WriteLine(new string('-', depath) + _name);
    }
}

它实现具体的叶节点的操作。

测试代码

class Program
{
    static void Main(string[] args)
    {
        Compnent rootCompnet = new Compsite("ROOT");

        Compnent rootLeaf1 = new Leaf("ROOTLEAF1");
        rootCompnet.Add(rootLeaf1);

        Compnent rootLeaf2 = new Leaf("ROOTLEAF2");
        rootCompnet.Add(rootLeaf2);

        Compnent rootCompsite= new Compsite("ROOTCOMPSITE");
        Compnent leaf1 = new Leaf("LEAF1");
        rootCompsite.Add(leaf1);
        Compnent leaf2 = new Leaf("LEAF2");
        rootCompsite.Add(leaf2);
        rootCompnet.Add(rootCompsite);

        rootCompnet.Display(0);

        Console.Read();
    }
}

个人理解

组合模式让客户端操作一个树形结构的对象，让客户端的操作统一起来。

适用性：

把对象表示成部分-整体的结构

用户忽略单个对象和组合对象的不同，用户统一的使用组合中的对象。

使得新增类型的组件更加容易。新定义的Composite或Leaf子类自动地与已有的结构和客户代码一起协同工作，客户程序不需因新的Component类而改变。

桥接模式

发表于 2015-08-23   |   分类于 设计模式 ， 结构型设计模式   |  

桥接模式(BridgePattern)是一种结构型设计模式。意图：将抽象部分与它的实现部分分离，使它们都可以独立地变化，从而将抽象与实现解耦。

代码实现

参与者

Abstraction： 1. 定义抽象接口。 2. 拥有一个实现（Implementor）类型对象引用。

RefinedAbstraction: 1. 扩展Abstraction中的接口定义。

public abstract class Shape
{
    protected Corlor _color;

    public abstract void DrawShape();

    public void SetColor(Corlor corlor)
    {
        _color = corlor;
    }
}

public class Rectangle : Shape
{
    public override void DrawShape()
    {
        _color.DrawColor();
        Console.WriteLine("长方形");
    }
}

public class Square : Shape
{
    public override void DrawShape()
    {
        _color.DrawColor();
        Console.WriteLine("正方形");
    } 
}

实现者

Implementor类定义了实现类的接口，这个接口不一定要与Abstraction的接口完全一致；实际上，这两个接口可以完全不同；

ConcreteImplementor类实现了Implementor定义的接口。

public abstract class Corlor
{
    public abstract void DrawColor();
}

public  class Red : Corlor
{
    public override void DrawColor()
    {
        Console.Write("红色的");
    }
}


public  class White : Corlor
{
    public override void DrawColor()
    {
        Console.Write("白色的");
    }
}

测试代码

class Program
{
    static void Main(string[] args)
    {
        Corlor corlor = new Red();
        Shape shape = new Square();

        shape.SetColor(corlor);
        shape.DrawShape();

        Console.Read();
    }
}

个人理解

桥接模式可以用来解决对象有2个或者2个以上的维度变化的问题。对客户隐藏抽象的实现部分（指的是实现者这部分）。

适用性：

你不希望在抽象和它的实现部分有一个固定的绑定关系。

类的抽象和它的实现部分都可以通过子类扩展。

对实现部分的修改，不会影响客户端（主要是指编译）。

如果某个变化在多个对象中共享是，可以抽象该变化，然后该变化的不同因素。

适配器模式

发表于 2015-08-22   |   分类于 设计模式 ， 结构型设计模式   |  

适配器模式(Adapter Pattern)是一种结构型设计模式。意图：将一个类的接口转换成客户希望的另外一个接口。Adapter模式使得原本由于接口不兼容而不能一起工作的那些类可以一起工作。

代码实现

一个接口提供外语的学习方法。原来是讲英语的，现在我们需要它讲法语。同样的接口做不同的工作。提供出来的新的功能的时候，我们可以用适配器模式

类适配器

public interface ITargert
{
    void Request();
}

public class Adaptee
{
    public void SpecialRequest()
    {
        Console.WriteLine("i can speek franch.");
    }
}

public class ClassAdapter : Adaptee, ITargert
{
    public void Request()
    {
        this.SpecialRequest();
    }
}

class Program
{
    static void Main(string[] args)
    {
        ITargert target = new ClassAdapter();
        target.Request();

        Console.Read();
    }
}

通过在类的Request()方法中我们调用Adaptee类的SpecialRequest()来实现法语。

对象适配器

public class Targert : ITargert
{
    public virtual void Request()
    {
        Console.WriteLine("i can speek english.");
    }
}

class ObjectAdapter : Targert
{
    private Adaptee _adaptee;

    public ObjectAdapter()
    {
        _adaptee = new Adaptee();
    }

    public override void Request()
    {
        _adaptee.SpecialRequest();
    }
}

class Program
{
    static void Main(string[] args)
    {
        ITargert targert = new ObjectAdapter();
        targert.Request();

        Console.Read();
    }
}

我们在ObjectAdapter中new一个Adaptee对象出来，然后在方法中Request()调用Adaptee对象的SpecialRequest()，来实现法语。

个人理解

类适配器和对象适配器都很好的复用了现有类。类适配器通过”多继承”的方式，使得它可以从新实现Adaptee的行为，也造成了高耦合的特点。对象适配器通过”对象组合”的方式，使得它松耦合，但是也不能重新定义Adaptee的行为，只能从Adaptee子类中实现。

适用性：

使用一个已经存在的类，而它的接口不符合你的需求。

创建一个可以复用的类，该类可以与其他不相关的类或不可预见的类（即那些接口可能不一定兼容的类）协同工作。

使用一些已经存在的子类，但是不可能对每一个都进行子类化以匹配它们的接口。对象适配器可以适配它的父类接口。

优缺点：

类适配器模式优点：1，可以在不修改原有代码的基础上来复用现有类，很好地符合 “开闭原则”。 2，可以重新定义Adaptee(被适配的类)的部分行为，因为在类适配器模式中，Adapter是Adaptee的子类。 3，仅仅引入一个对象，并不需要额外的字段来引用Adaptee实例（这个即是优点也是缺点）。

类适配器模式缺点：1，用一个具体的Adapter类对Adaptee和Target进行匹配，当如果想要匹配一个类以及所有它的子类时，类的适配器模式就不能胜任了。因为类的适配器模式中没有引入Adaptee的实例，光调用this.SpecificRequest方法并不能去调用它对应子类的SpecificRequest方法。 2， 采用了 “多继承”的实现方式，带来了不良的高耦合。

对象适配器的优点:1, 可以在不修改原有代码的基础上来复用现有类，很好地符合 “开闭原则”（这点是两种实现方式都具有的）。 2，采用 “对象组合”的方式，更符合松耦合。

对象适配器的缺点:1，使得重定义Adaptee的行为较困难，这就需要生成Adaptee的子类并且使得Adapter引用这个子类而不是引用Adaptee本身。

原型模式

发表于 2015-08-21   |   分类于 设计模式 ， 创建型设计模式   |  

原型模式（Prototype Pattern）是一种创建型设计模式。意图：用原型实例指定创建对象的种类，并且通过拷贝这些原型创建新的对象。

代码实现

对象类

[Serializable]
public class SalesDepartment
{
    public string _manager;
    public string _employee_A;
    public string _employee_B;

    public SalesDepartment(string manager, string employee1, string employee2)
    {
        _manager = manager;
        _employee_A = employee1;
        _employee_B = employee2;
    }
}

[Serializable]
public class FinanceDepartment
{
    public string _manager;
    public string _employee_C;
    public string _employee_D;

    public FinanceDepartment(string manager, string employee1, string employee2)
    {
        _manager = manager;
        _employee_C = employee1;
        _employee_D = employee2;
    }

}

[Serializable]
public class Company
{
    //总裁
    public string _ceo;
    //经理
    public string _manager;
    //销售部
    public SalesDepartment _salesDepartMent;
    //财务部
    public FinanceDepartment _finaceDepartMent;

    public Company(string ceo, string manager)
    {
        _ceo = ceo;
        _manager = manager;

        _salesDepartMent = new SalesDepartment("张三", "员工sales1", "员工sales1");
        _finaceDepartMent = new FinanceDepartment("李四", "员工finace", "员工finace1");
    }

    public Company clone(bool isDeepClone)
    {
        Company company = null;
        if (isDeepClone)
        {
            MemoryStream memoryStream = new MemoryStream();
            BinaryFormatter formatter = new BinaryFormatter();
            formatter.Serialize(memoryStream, this);
            memoryStream.Position = 0;
            company = (Company)formatter.Deserialize(memoryStream);
        }
        else
        {
            company = (Company)this.MemberwiseClone();
        }

        return company;
    }
}

测试部分代码

static void Main(string[] args)
{
    Company test1 = new Company("ceo", "manager");
    Company test2 = test1.clone(false);

    test1._finaceDepartMent._manager = "赵六";

    Console.WriteLine(test1._ceo.Equals(test2._ceo)? test1._ceo: (test1._ceo+ "" + test2._ceo));

    string manager1, manager2;
    manager1 = test1._finaceDepartMent._manager;
    manager2 = test1._finaceDepartMent._manager;
    Console.WriteLine(manager1.Equals(manager2) ? manager1 : (manager1 + "" + manager2));

    Company test3 = new Company("ceo3", "manager4");
    Company test4 = test3.clone(true);

    test3._finaceDepartMent._manager = "赵六";

    Console.WriteLine(test1._ceo.Equals(test2._ceo) ? test1._ceo : (test1._ceo + "" + test2._ceo));

    manager1 = test3._finaceDepartMent._manager;
    manager2 = test4._finaceDepartMent._manager;
    Console.WriteLine(manager1.Equals(manager2) ? manager1 : (manager1 + "" + manager2));

    Console.Read();
}

prototype原型要实现一个clone的方法来拷贝自身对象。

在Company类中，实现了一个深拷贝和一个浅拷贝。通过测试我们发现深拷贝的时候，当财务经理变化的时候，拷贝类的财务经理并没有变化。而浅拷贝的类在原类的财务经理变化的时候，拷贝类的财务经理发生了变化。我们可以理解为，浅拷贝只是给原来的公司换了个名字，深拷贝是按照原来的公司的结构创建了另外的一家相同结构的公司。

个人理解

原型是一个对原类拷贝，但是它并不调用构造函数。比起用new来创建对象，在效率上会提高。但是原型模式会出现深拷贝和浅拷贝的问题，对于基本类型都是值拷贝，对于数组，容器，对象等都是引用拷贝，如果要实现值拷贝，就要自己new出各个类的手工实例或者通过序列化和反序列化来实现拷贝。

适用性：

当要实例化的类是在运行时刻指定时，例如，通过动态装载。

为了避免创建一个与产品类层次平行的工厂类层次时。

当一个类的实例只能有几个不同状态组合中的一种时。建立相应数目的原型并克隆它们可能比每次用合适的状态手工实例化该类更方便一些。

如果你创建的对象是多变化、多等级的产品，或者产品的创建过程非常耗时的时候(比如，有一定的计算量，或者对象创建时需要从网络或数据库中获取一定的数据)，或者想把产品的创建独立出去，不想了解产品创建细节的时候可以考虑使用。不得不说，原型模式给了我们多一种创建对象，并且不依赖具体对象的选择。

特点：

对客户端隐藏了具体的产品类。

允许客户只通过注册原型实例就可以将一个具体产品类并入到系统中，客户可以在运行时刻建立和删除原型；

减少了子类的构造。原型模式是Clone一个原型而不是请求工厂方法创建一个，所以它不需要一个与具体产品类平行的Creator类层次。

原型模式具有给一个应用软件动态加载新功能的能力。由于Prototype的独立性较高，可以很容易动态加载新功能而不影响旧系统；

产品类不需要非得有任何事先确定的等级结构，因为原型模式适用于任何的等级结构。

原型模式的最重要缺点就是每一个类必须配备一个Clone方法，而且这个Clone方法需要对类的功能进行通盘考虑。这对全新的类来说不是很难，但对已有的类进行改造时，不一定是容易的事。

建造者模式

发表于 2015-08-20   |   分类于 设计模式 ， 创建型设计模式   |  

建造者模式(BuilderPattern)是一种创建型设计模式。意图：将一个复杂对象的构建与它的表示分离，使得同样的构建过程可以创建不同的表示。

产品类

class Product
{
    private string _partHead { get; set; }
    private string _partBody { get; set; }
    private string _partFoot { get; set; }

    public void setHead(string property)
    {
        _partHead = property;
    }

    public void setBody(string property)
    {
        _partBody = property;
    }

    public void setFoot(string property)
    {
        _partFoot = property;
    }


    public void show() 
    {
        Console.WriteLine("Head :" + _partHead);
        Console.WriteLine("Body :" + _partBody);
        Console.WriteLine("Foot :" + _partFoot);
    }
}

建造类

abstract class Builder
{        
    public abstract void BuildHead();

    public abstract void BuildBody();

    public abstract void BuildFoot();

    public abstract Product GetProduct();
}

class ConcretBuilderAnimal : Builder
{
    private Product _product;

    public ConcretBuilderAnimal()
    {
        _product = new Product();

    }
    public override void BuildHead()
    {
        _product.setHead("this is animal head!");
    }

    public override void BuildBody()
    {
        _product.setBody("this is animal body!");
    }

    public override void BuildFoot()
    {
        _product.setFoot("this is animal foot!");
    }

    public override Product GetProduct()
    {
        return _product;
    }
}

class ConcretBuilderMan : Builder
{
    private Product _product;

    public ConcretBuilderMan()
    {
        _product = new Product();
    }

    public override void BuildHead()
    {
        _product.setHead("this is man head!");
    }

    public override void BuildBody()
    {
        _product.setBody("this is man body!");
    }

    public override void BuildFoot()
    {
        _product.setFoot("this is man foot!");
    }

    public override Product GetProduct()
    {
        return _product;
    }
}

导演类

class Director
{
    public void Construct(Builder builder)
    {
        builder.BuildHead();
        builder.BuildBody();
        builder.BuildFoot();
    }
}

具体测试和实现

class Program
{
    static void Main(string[] args)
    {
        Builder builder = new ConcretBuilderMan();
        Director director = new Director();
        director.Construct(builder);

        Product product = builder.GetProduct();
        product.show();

        Console.ReadLine();
    }
}

建造类负责构造对象的细节，由导演类统一的规范流程来构造对象。使得同样的流程可以建造出不同的对象。

个人理解

举汽车建造过程的例子来说明：建造者生产不同汽车的轮胎，外壳， 系统。然后将这些产品拿到外面的工厂按照统一的组装成汽车。但是出来的汽车可能是甲壳虫，奥迪等。该模式的特点是：建造对象的算法，既独立于对象的组成部分，也独立于对象的装配过程。工厂模式中， 对象装配过程也在对象中。两者的使用区别是：构造复杂的对象，而且客户端不必知道产品的细节的时候，用工厂模式，构造更加复杂的对象用建造者模式

好处：

客户端不必知道产品的细节。

建造者相互独立，有利于系统扩展。

建造者是独立的，可以对建造的过程进行细化，而不影响其他的模块。

适用范围：

构建复杂的对象，构建的流程是统一的，构建的组件是复杂的。

构建对象的算法，既独立于构建过程，也独立于组成部分的装配方法。

单例模式

发表于 2015-08-19   |   分类于 设计模式 ， 创建型设计模式   |  

单例模式(Singleton Pattern)是一种创建型设计模式。意图：保证一个类仅有一个实例，并提供一个访问它的全局访问点。

懒汉式

public class SingletonA
{
    private static SingletonA _singleton;
    private SingletonA()
    {
    }

    public static SingletonA GetInstance()
    {
        if (null == _singleton)
        {
            _singleton = new SingletonA();
        }

        return _singleton;
    }
}

static void Main(string[] args)
{
    SingletonA a1 = SingletonA.GetInstance();
    SingletonA a2 = SingletonA.GetInstance();
    if (a1.Equals(a2))
    {
        Console.WriteLine("a1 = a2");
    }
    else
    {
        Console.WriteLine("a1 != a2");
    }

    Console.ReadLine();
}

饿汉式

public class SingletonB
{
    private static SingletonB _singleton = new SingletonB();
    private SingletonB()
    {
    }

    public static SingletonB GetInstance()
    {
        return _singleton;
    }
}

static void Main(string[] args)
{
    SingletonB b1 = SingletonB.GetInstance();
    SingletonB b2 = SingletonB.GetInstance();
    if (b1.Equals(b2))
    {
        Console.WriteLine("b1 = b2");
    }
    else
    {
        Console.WriteLine("b1 != b2");
    }

    Console.ReadLine();
}

双重检查锁定

public class SingletonLock
{
    private static SingletonLock _singleton;
    private static readonly Object sysncObject = new Object();

    private SingletonLock()
    {
    }

    public static SingletonLock GetInstance()
    {
        if (null == _singleton)
        {
            lock (sysncObject)
            {
                if(null == _singleton) {
                    _singleton = new SingletonLock();
                }
            }
        }
        return _singleton;
    }

}

static void Main(string[] args)
{    
    SingletonLock lock1 = SingletonLock.GetInstance();
    SingletonLock lock2 = SingletonLock.GetInstance();
    if (lock1.Equals(lock2))
    {
        Console.WriteLine("lock1 = lock2");
    }
    else
    {
        Console.WriteLine("lock1 != lock2");
    }

    Console.ReadLine();
}

适用性

适用性 整个应用程序的生命周期只需要一个实例。

各种写法的比较 懒汉式适用于单线程的程序，在多线程的程序中会造成多个实例。 双重锁检查的写法可以很好的解决多线程的问题。饿汉式的写法在类装载的时候就出事话，没有达到懒汉式装载的效果。

工厂模式

发表于 2015-08-19   |   分类于 设计模式 ， 创建型设计模式   |  

工厂模式(Factory Pattern)是一种创建型设计模式。意图：定义一个用于创建对象的接口，让子类决定实例化哪一个类。Factory Method 使一个类的实例化延迟到其子类。

简单工厂模式

public class Product
{
    public virtual void show()
    {
    }

    public string className()
    {
        return this.GetType().Name;
    }
}

public class ProductA : Product
{
    public override void show()
    {
        Console.WriteLine("this is " + className());
    }
}

public class ProductB : Product
{
    public override void show()
    {
        Console.WriteLine("this is " + className());
    }
}

enum ProductType {
    A = 0,
    B,
    UNKONW= -1
}

class SimpleFactory
{
    public Product createProduct(ProductType type)
    {
        Product product = null;
        switch (type)
        {
        case ProductType.A:
            product = new ProductA();
            break;
        case ProductType.B:
            product = new ProductB();
            break;
        }

        return product;
    }
}

static void Main(string[] args)
{
    SimpleFactory factory = new SimpleFactory();

    Product product = factory.createProduct(ProductType.A);
    product.show();

    Console.ReadLine();
}

工厂模式

public class Factory
{
    public virtual Product createProduct()
    {
        return null;
    }
}

public class FactoryA : Factory
{
    public override Product createProduct()
    {
        return new ProductA();
    }
}

public class FactoryB : Factory
{
    public override Product createProduct()
    {
        return new ProductB();
    }
}

static void Main(string[] args)
{
    Factory factory = new FactoryB();
    Product product = factory.createProduct();
    product.show();
}

抽象工厂模式

public class AbstractFactory
{
    public virtual Product createProductA()
    {
        return null;
    }

    public virtual Product createProductB()
    {
        return null;
    }
}

public class AbstractFactoryA : AbstractFactory
{
    public override Product createProductA()
    {
        return new ProductA1();
    }

    public override Product createProductB()
    {
        return new ProductB1();
    }
}

public class AbstractFactoryB : AbstractFactory
{
    public override Product createProductA()
    {
        return new ProductA();
    }

    public override Product createProductB()
    {
        return new ProductB();
    }

}

public class ProductA1 : Product
{
    public override void show()
    {
        Console.WriteLine("this is " + className());
    }
}

public class ProductB1 : Product
{
    public override void show()
    {
        Console.WriteLine("this is " + className());
    }
}

static void Main(string[] args)
{
    AbstractFactory factory = new AbstractFactoryA();
    Product product = factory.createProductA();
    product.show();
}

优缺点

适用性：类的实例由子类指定。

优缺点：简单工厂模式实现了责任的分离，也造成了工厂类的复杂性，每添加一个产品都必须修改工厂类，不符合开放和封闭原则。工厂模式
解决了简单工厂模式工厂类的复杂性问题，但是产品多样的时候，工厂类众多。抽象工厂模式实现了一类产品的构造。
